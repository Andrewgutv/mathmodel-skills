# Problem Solving Loop

## Purpose
当任务是一个真实数学建模项目时，必须完成下面这条闭环：

1. 读题
2. 拆题
3. 选模
4. 写可运行代码
5. 生成真实结果
6. 生成图表
7. 把结果同步回论文、说明文档和答辩材料

这不是“只写论文”的流程，也不是“只给方案”的流程。

## Standard Delivery Target
- 默认优先交付 Word 版本，而不是只给 Markdown。
- 如果项目目录中已有老师模板，默认要尽量生成模板一致版。
- 如果只有 `.doc` 模板，也应保留这一依赖，并尽量转换到最接近的可复现 `docx` 流程。
- 如果用户没有指定页数要求，不要机械追求篇幅，优先保证结构完整、结果真实、图表够用。

## Project Pattern
新项目默认收敛为以下结构：

- `代码/`
- `图片/`
- `数据/`
- `结果/`
- `文档/`
- `答辩/`

不要在项目里大量散落“临时版”“修正版”“v2”脚本与文档；应尽量收敛为稳定主产物。

## Step 1: Read the Prompt and Inventory Inputs
- 先读题目主文件，再读附件与数据说明。
- 按 `references/prompt-parsing.md` 识别目标、约束、已知量、未知量、输出要求。
- 盘点所有输入文件，并登记到 `数据/数据说明表.md`。
- 多个问题时，明确标为 `问题1`、`问题2`、`问题3`。
- 如果 PDF 是扫描版或整页图片，必须启用 OCR fallback；不能把“抽不出文本”误判成“题面没有问题”。

本步至少产出：

- `文档/方案.md`
- `文档/解题思路.md`
- `数据/数据说明表.md`

## Step 2: Split the Problem
对每个子问题都要写清楚：

- 目标
- 输入
- 输出
- 约束
- 候选方法
- 选定方法

同时生成或更新：

- `文档/方案.md`
- `文档/解题思路.md`
- `文档/问题-方法对照表.md`

不要用一个含糊的大模型覆盖所有问题，除非题目本身就支持这样做。

## Step 3: Design the Code Layout
编码前先定代码布局。推荐至少包括：

- `代码/main.py` 或 `代码/run_all.py`
- `代码/preprocess.py`
- `代码/model_utils.py`
- `代码/code_问题1_xxx.py`
- `代码/code_问题2_xxx.py`
- `代码/code_问题3_xxx.py`
- `代码/plot_figures.py`
- `代码/export_results.py`

如果项目规模较小，也至少要做到“子问题和代码文件之间一一对应”。

## Step 4: Implement Real Models
- 必须写真实 Python 代码，不写伪代码冒充实现。
- 必须优先读取真实数据；如果数据缺失，必须显式写出假设。
- 每个主要脚本都要明确：
  - 读什么
  - 算什么
  - 写什么
  - 生成什么图

本步至少产出：

- `代码/` 下的可运行脚本
- `文档/解题过程.md` 的初版

## Step 5: Run or Validate the Computation Path
- 环境允许时应实际运行。
- 若不能完整运行，必须准确写清哪些步骤已验证、哪些未运行、原因是什么。
- 机器可读结果必须进入 `结果/`。
- 运行说明和关键信息进入 `结果/运行日志.md`。

推荐结果命名：

- `结果/问题1_结果汇总.csv`
- `结果/问题2_结果汇总.csv`
- `结果/问题3_结果汇总.csv`
- `结果/综合结果汇总.csv`

不要在论文里引用不存在于 `结果/` 的数字。

## Step 6: Generate Real Figures
- 图必须由真实结果生成，不能用装饰性占位图。
- 图文件进入 `图片/`。
- 同时维护 `图片/图表清单.md`，记录：
  - 文件名
  - 对应问题
  - 用途
  - 是否已被正文引用

## Step 7: Sync Results Back Into Documents
在代码、结果、图件到位后，再同步回文档。

### 论文
更新 `文档/论文.md`，确保：

- 表格来自 `结果/`
- 图片来自 `图片/`
- 参数来自真实代码或结果
- 结论被数据支持

### 解题过程
更新 `文档/解题过程.md`，说明：

- 跑了哪些脚本
- 用了哪些数据
- 生成了哪些结果
- 做了哪些敏感性分析

### 模型解释
更新：

- `文档/模型解释_小白版.md`
- `文档/模型解释_专业版.md`

两者解释的是同一套模型，只是抽象层不同。

### 答辩材料
若需要答辩，更新：

- `答辩/答辩准备.md`
- `答辩/答辩讲稿.md`
- `答辩/PPT设计文档.md`
- `答辩/老师高频问题与回答.md`

## Step 8: Build DOCX
Word 交付推荐走固定流水线：

1. 规范标题结构
2. 生成 `文档/论文_完整终稿.docx`
3. 如存在老师模板，再生成 `文档/论文_模板一致版.docx`
4. 刷新目录、字段、分页
5. 检查是否存在重复摘要、重复目录、重复前置页残留

如果模板版失败但完整终稿成功，必须：

- 保留 `文档/论文_完整终稿.docx`
- 在运行日志或流水线报告中明确记录模板版失败原因

更细流程参考 `references/word-pipeline.md`。

## Step 9: Validate Before Delivery
交付前至少检查：

- 是否存在最终论文 Markdown
- 是否存在完整终稿 DOCX
- 如有模板，是否存在模板一致版 DOCX
- 关键结果文件是否存在
- 核心图表是否存在并被正文引用
- 答辩材料是否和论文一致

如果已经有 `scripts/check_paper.py` 可用，优先跑它；否则做等价人工检查。
