# Workflow Reference

## Goal
把一个原始数学建模题目收敛为一个可复现、可交付、可检查的项目包，而不是零散的脚本和文档。

默认统一入口：

- 项目内：`run_mathmodel.ps1`
- 技能脚本：`scripts/run_pipeline.py`

默认阶段顺序：

- `parse`
- `solve`
- `summarize`
- `write`
- `word`
- `check`

完整交付目标默认包括：

- `代码/`
- `图片/`
- `数据/`
- `结果/`
- `文档/方案.md`
- `文档/解题思路.md`
- `文档/解题过程.md`
- `文档/模型解释_小白版.md`
- `文档/模型解释_专业版.md`
- `文档/论文.md`
- `文档/论文_完整终稿.docx`
- `文档/论文_模板一致版.docx`（当提供老师模板时）
- `答辩/答辩准备.md`
- `答辩/答辩讲稿.md`
- `答辩/PPT设计文档.md`
- `答辩/老师高频问题与回答.md`

更细的执行顺序，始终同时参考 `references/problem-solving-loop.md`。

## Phase 1: Read and Analyze
最少应产出：

- `文档/方案.md`
- `文档/解题思路.md`
- `文档/问题-方法对照表.md`
- `数据/数据说明表.md`

要求：

- 明确题目重述
- 明确子问题拆分
- 明确候选方法与最终选型
- 明确所需数据、已知约束和风险项
- 输出必须足以指导后续编码，而不是停留在泛泛讨论

## Phase 2: Model and Visualize
最少应做到：

- 每个主要子问题有对应脚本或清晰的实现模块
- 结果落到 `结果/`
- 图件落到 `图片/`
- `文档/解题过程.md` 记录真实运行和产物

推荐输出：

- `代码/run_all.py` 或统一入口脚本
- `结果/运行日志.md`
- `图片/图表清单.md`

## Phase 3: Write the Paper
最少应做到：

- 按竞赛论文结构生成 `文档/论文.md`
- 符号先定义后使用
- 图、表、公式在正文中有引用
- 摘要覆盖问题、方法、结果、贡献
- 所有数值都来自 `结果/`

同时应生成：

- `文档/模型解释_小白版.md`
- `文档/模型解释_专业版.md`

这两个文档解释的是同一套模型，只是面向不同读者。

## Phase 4: Generate DOCX
最少应做到：

- 生成独立的 Word 成品，不覆盖源 Markdown
- 标题层级与老师模板顺序一致
- 如果老师模板存在，则尽量保留封面、承诺书、目录与前置页
- 明确区分：
  - `文档/论文_完整终稿.docx`
  - `文档/论文_模板一致版.docx`

## Phase 5: Prepare the Defense
最少应做到：

- 答辩准备材料存在
- 讲稿存在
- PPT 设计文案存在
- 常见问答存在
- 答辩材料的数字和结论与论文一致

## Non-Goals
以下情况不应被误当成完整流程完成：

- 只写了方案，没有代码
- 只写了 Markdown，没有结果文件
- 只有图，没有结果表
- 只有 Word 外壳，没有完整正文
- 只有答辩文案，没有回溯到论文和结果
