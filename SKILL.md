# MathModel

## Scope
- 面向中文数学建模竞赛或课程大作业，不按通用学术论文技能处理。
- 默认目标是“可提交的竞赛论文与配套材料”，不是只写思路或只给模型建议。
- 当用户只要求某一阶段时，只执行该阶段，不强行展开全流程。
- 当任务是完整建模项目时，必须覆盖：读题、拆题、选模、写代码、生成结果、写论文、交付 Word、整理答辩材料。
- 优先适配 Windows + PowerShell + Python 的工作环境。
- 老师模板、封面页、承诺书、学校要求页等都视为外部交付依赖；如果已提供，则必须尽量保留。

## Default Entry
- 默认入口是 `/mathmodel <项目路径>`。
- 如果用户没有显式使用 `/mathmodel`，但需求明显属于数学建模全流程，也按本技能执行。
- 项目内一键入口优先使用 `run_mathmodel.ps1`。
- 技能内部统一调度脚本优先使用 `scripts/run_pipeline.py`。
- 默认项目路径下应至少包含：
  - 题目文件：PDF、DOCX、Markdown 或图片之一
  - 附件数据：Excel、CSV、图片、说明文档等
  - 如需模板交付：老师模板 `docx` 或 `doc`

## Project Structure
完整项目默认收敛为以下目录：

- `代码/`
- `图片/`
- `数据/`
- `结果/`
- `文档/`
- `答辩/`

约定如下：

- `代码/`：可运行脚本、预处理脚本、绘图脚本、统一入口脚本
- `图片/`：正文图、敏感性分析图、结构示意图、流程图
- `数据/`：原始题目附件、整理后的中间数据、数据说明
- `结果/`：CSV、TXT、XLSX 等机器可读结果，以及运行日志
- `文档/`：方案、解题思路、解题过程、论文 Markdown、最终 Word
- `答辩/`：答辩准备、PPT 文案、讲稿、老师高频问题

如果用户只做单阶段任务，不强制创建全部目录，但新产物仍应尽量归入上述结构。

## Required Inputs
- 题目文件：优先用户明确指定的文件；未指定时优先 PDF，其次 DOCX，再次图片，最后 Markdown。
- 数据文件：应与题目位于同级目录或明确子目录中。
- 模板文件：当用户要求“老师模板版”或“模板一致版”时必须存在。
- 对扫描版 PDF 或图片型 PDF，必须允许 OCR fallback；不能因为 `pdfplumber`/`pypdf` 抽不出文本就把题面识别为 0 问。

## Standard Outputs
完整流程下，至少应产出以下文件：

- `文档/方案.md`
- `文档/解题思路.md`
- `文档/解题过程.md`
- `文档/论文.md`
- `文档/模型解释_小白版.md`
- `文档/模型解释_专业版.md`
- `答辩/答辩准备.md`
- `答辩/答辩讲稿.md`
- `答辩/PPT设计文档.md`

如执行 Word 交付，还应产出：

- `文档/论文_完整终稿.docx`
- `文档/论文_模板一致版.docx`

如执行计算与图表生成，还应产出：

- `结果/` 下的机器可读结果文件
- `图片/` 下的正式图件

## Workflow

### Phase 1: 读题与规划
必须完成：

1. 读取题目与附件，识别所有子问题、约束、输入和目标。
2. 盘点可用数据与缺失信息。
3. 形成明确的子问题拆分，不允许只写笼统思路。
4. 写出 `文档/方案.md`，至少包含：
   - 题目重述
   - 子问题拆解
   - 候选方法
   - 选定方法与理由
   - 所需数据
   - 风险与不确定项
5. 同步生成：
   - `文档/解题思路.md`
   - `文档/问题-方法对照表.md`
   - `数据/数据说明表.md`

这一阶段不能停留在“分析完了”。输出必须能直接驱动后续代码实现。

### Phase 2: 建模、编码、出结果
必须完成：

1. 根据子问题划分代码结构。
2. 为主要子问题编写真正可运行的 Python 脚本。
3. 明确每个脚本：
   - 读取什么数据
   - 写出什么结果
   - 生成什么图片
4. 在环境允许时实际运行；若不能完整运行，必须明确卡点。
5. 将机器可读结果保存到 `结果/`。
6. 将图件保存到 `图片/`。
7. 更新 `文档/解题过程.md`，记录：
   - 跑了哪些脚本
   - 用了哪些数据
   - 得到了哪些输出
8. 关键参数或假设会明显影响结果时，必须补做敏感性分析。

### Phase 3: 撰写论文
必须完成：

1. 按竞赛论文结构撰写 `文档/论文.md`。
2. 符号必须先定义后使用。
3. 所有核心数值必须来自 `结果/`，不能凭空编。
4. 图、表、公式必须在正文中被引用。
5. 摘要最后写，且必须覆盖：
   - 问题
   - 方法
   - 结果
   - 贡献
6. 如要求 AI 使用声明，必须在文末补齐。

### Phase 4: 生成 Word
默认优先走可复现流水线，而不是手工在 Word 里一点点改。

必须完成：

1. 规范一级标题顺序，确保与老师模板一致。
2. 以 `文档/论文.md` 或正文 `docx` 为基础生成：
   - `文档/论文_完整终稿.docx`
3. 如果有老师模板，则进一步生成：
   - `文档/论文_模板一致版.docx`
4. 若模板带封面、承诺书、目录或前置页，应尽量保留。
5. 输出文件名必须与源文件区分，避免覆盖已打开文件。
6. 若模板合并阶段失败，但 `论文_完整终稿.docx` 已成功生成，则必须保留完整终稿并明确报告模板版失败原因。

### Phase 5: 答辩材料
如用户要求答辩或完整交付，必须完成：

1. `答辩/答辩准备.md`
2. `答辩/答辩讲稿.md`
3. `答辩/PPT设计文档.md`
4. `答辩/老师高频问题与回答.md`

并确保其数值与结论和 `结果/`、`文档/论文.md` 保持一致。

## Pipeline Stages
项目级主入口 `scripts/run_pipeline.py` 与 `run_mathmodel.ps1` 应按以下阶段执行：

- `parse`
- `solve`
- `summarize`
- `write`
- `word`
- `check`

含义如下：

- `parse`：解析题目、更新方案和解题思路
- `solve`：运行 `代码/` 下的求解脚本并产出 `结果/`
- `summarize`：把各问题结果聚合为 `结果/综合结果汇总.csv`
- `write`：把 `结果/` 回填到论文和答辩文档
- `word`：生成 `论文_完整终稿.docx` 和可选的 `论文_模板一致版.docx`
- `check`：检查代码、结果、文档、Word 交付是否一致

## Decision Rules

### Must Do
- 必须真正读题、拆题、选模、写代码，而不是停在空泛建议。
- 必须产出真实结果后再写结论。
- 必须保持代码、结果、论文、答辩材料之间一致。
- 必须优先保留老师要求的标题体系和前置页结构。
- 必须把机器可读结果与正式论文分开存放：结果进 `结果/`，正文进 `文档/`。
- 必须在完整项目中保留可追溯链路：题目 -> 数据 -> 脚本 -> 结果 -> 论文 -> Word。
- 必须把“扫描版 PDF 识别失败”和“没有 solver”区分开，不允许把 OCR 问题误报成题目没有子问题。

### Should Do
- 应在有意义时补做敏感性分析或方法对比。
- 应使用稳定文件名，避免同一产物出现大量 `v2`、`修正版`、`终稿2`。
- 应在关键阶段留下运行记录与交付检查结果。

### Do Only When Applicable
- 只有在用户要求模板版时，才需要模板灌入。
- 只有在用户要求答辩材料时，才需要生成答辩相关文档。
- 只有在存在多个可信方法时，才需要方法对比。

## Validation Checklist

### 项目级检查
- [ ] 目录结构清晰，代码/图片/数据/结果/文档/答辩分离
- [ ] 每个子问题都有对应方法、代码或明确处理说明
- [ ] 关键结果已落到 `结果/`
- [ ] 论文中的核心数字可追溯到结果文件
- [ ] 图表文件存在，且正文已引用

### Word 交付检查
- [ ] `文档/论文_完整终稿.docx` 可打开
- [ ] 如有模板，`文档/论文_模板一致版.docx` 可打开
- [ ] 模板版保留封面、承诺书、目录等前置页
- [ ] 没有重复摘要、重复目录、重复封面残留

### 答辩材料检查
- [ ] 答辩准备、讲稿、PPT 文案存在
- [ ] 答辩材料中的数字与论文一致
- [ ] 已准备模型优点、缺点、假设、参数选择的回答

## Public Boundary
- 这是一个面向中文数学建模竞赛的专用技能，不应描述成通用科研写作技能。
- 它能显著加速完整项目交付，但不保证对任意学校模板都“一键零人工”。
- 当模板、字体、Word 环境或题目附件缺失时，必须明确指出依赖，再给出最接近的可复现交付结果。
